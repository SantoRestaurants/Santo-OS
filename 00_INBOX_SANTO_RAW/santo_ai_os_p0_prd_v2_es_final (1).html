Resumen ejecutivo

Santo AI OS P0 v2 es la primera versión construible de la capa de inteligencia operativa de Santo. Prueba el modelo OS con un workflow primario, Corte Santo, y dos validaciones secundarias delgadas: Validación XML SAT y Recibos de Utilidades.

Esta v2 corregida integra directamente los últimos ajustes de auditoría. No habrá addendum técnico separado por ahora. El único addendum posterior debe ser con respuestas operativas confirmadas por Manuela, Abraham y el equipo admin.

Principio operativo P0

P0 prueba el modelo operativo, no toda la visión. Un dominio, un workflow primario, una unidad primero, un dashboard, Agent Mail como canal controlado de entrada/notificación, un modelo de revisión/aprobación y una columna vertebral de registros.

Agent Mail en P0

Agent Mail, usando una dirección como os@santo.com, es un componente real de P0. Es el canal controlado de entrada y notificación. El sistema no debe necesitar acceso amplio al inbox de Manuela para iniciar; los insumos se mandan o reenvían al inbox del OS.

No objetivos P0

  • No WhatsApp productivo.
  • No motor completo de reporting financiero.
  • No automatización SAT/FIEL/IDSE/bancos/nómina.
  • No aprobaciones autónomas de alto riesgo.
  • No hardcodear rutas de Drive, thresholds, revisores, routing de emails o supuestos de restaurantes/entidades.
  • No reorganización completa de Drive.
  • No escritura a Sheets para Utilidades salvo confirmación explícita.

Insumos del equipo que serán el único addendum futuro

  • URLs, jerarquía y naming de Drive.
  • Thresholds, severidades y adjuntos obligatorios de Corte.
  • Mapa de revisores por tipo de excepción.
  • Mapeos restaurante/entidad/RFC y códigos cortos.
  • Convención de routing en Agent Mail: prefijos, labels, remitentes o reglas.
  • Columnas de plantilla de utilidades y reglas de marcado/color.
  • Confirmar si Utilidades puede escribir en Google Sheets en P0 o solo registrar estatus en Supabase.
  • Al menos un XML exportado real anonimizado o sanitizado desde MiAdminXML para crear fixtures de prueba.

Criterios de éxito

  • Un email al inbox OS se parsea, crea email_messages y se liga a workflow_run cuando se clasifica.
  • Corte Santo crea workflow_run para un restaurante/fecha.
  • Documentos/evidencia se ligan a workflow_run/tasks.
  • Inputs faltantes o thresholds no confirmados quedan como requires_review, nunca pasan silenciosamente.
  • Dashboard muestra Corte, runs, excepciones y revisiones.
  • Agent Mail envía resumen o alerta.
  • Todas las acciones generan eventos/auditoría.
  • Re-ejecutar el mismo input no duplica ni corrompe datos.
  • Workflows secundarios reutilizan la misma foundation.
