Propósito y regla de construcción

Esta Especificación Técnica v2 corregida es la guía paso a paso para Dante en P0. Reemplaza la v2 anterior. No se debe crear addendum técnico separado; el único addendum posterior debe contener respuestas operativas confirmadas por el equipo Santo.

Regla crítica: Dante puede iniciar PR 1–PR 5 de inmediato. Dante no debe hardcodear rutas de Drive, thresholds, revisores, códigos de restaurante, routing de emails, source_hash ni supuestos de negocio. Todo debe ser configurable y confirmado por Santo.

      "idempotency_key": "string",
      "created_documents": [],
      "created_tasks": [],
      "created_exceptions": [],
      "approvals_required": [],

      "errors": []
    }
    """
  • Debe soportar dry_run.
  • Debe crear o reutilizar workflow_runs vía idempotency.
  • No debe pasar silenciosamente si falta config.
  • Debe loggear start/end/failure en watchdog.
  • Debe devolver requires_review cuando thresholds/reglas estén pendientes.

Comportamiento default si faltan reglas

  • Thresholds no confirmados: cada check genera excepción media y workflow_run queda requires_review.
  • Folder no confirmado: no escribir en carpeta desconocida; crear placeholder y excepción.
  • Reviewer no confirmado: usar owner default si existe; si no, requires_review.
  • Email no clasificado: email_messages.processing_status = requires_review; no adivinar.
  • Ningún threshold pendiente debe pasar como completed.

Routing de Agent Mail

La convención final está pendiente. Debe ser configurable.

OpciónEjemploNotas
Prefijo asunto[CORTE], [XML], [UTILIDADES]Recomendado para P0 si el equipo puede cumplir.
Por remitenteEmail gerente → CorteÚtil pero riesgoso si mezclan contenido.
Gmail labelLabel manual o reglaBuen puente si asuntos son inconsistentes.
Tipo adjunto.xml → XMLSeñal de apoyo, no autoridad única.
ignored vs requires_review: ignored solo puede usarse cuando el correo es definitivamente no-workflow por una regla confirmada, por ejemplo newsletter automática, spam conocido o remitente explícitamente excluido. Todo correo ambiguo debe quedar como requires_review, nunca ignored.

Criterios de aceptación antes de merge

  • No secrets en repo.
  • RLS activo.
  • Service role no expuesto al frontend.
  • workflow_runs y email_messages explícitos.
  • Cada módulo tiene skill.md, config.example.json y una sección explícita de Dependencias de Modelo de Datos con tablas leídas/escritas.
  • Cada script soporta dry_run e idempotency.
  • Reglas pendientes default a requires_review.
  • Agent Mail no adivina routing.
  • Dashboard muestra runs y excepciones.
  • Tests cubren duplicados/idempotency y falta de config.
