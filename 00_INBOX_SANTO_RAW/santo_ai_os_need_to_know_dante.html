Santo AI OS · Need-to-know para Dante

Qué cambió, por qué importa y cómo debe construir P0

Documento corto para explicar los ajustes recientes sin incluir el contexto interno de la conversación.

1. La idea central

Dante, ajustamos las instrucciones para que el proyecto no se convierta en una colección de scripts sueltos.

Santo AI OS debe ser una base común donde cada workflow fortalece la memoria operativa de Santo.

Principio: si una automatización funciona, pero no se conecta a la columna vertebral del OS, no está bien construida para este proyecto.

Todos los workflows deben conectarse a la misma estructura base:

  • workflows
  • workflow_runs
  • documents
  • tasks
  • exceptions
  • approvals/reviews
  • events
  • watchdog_log
  • drive_folder_map
  • email_messages

2. Por qué lo estructuramos así

Tomamos una lógica tipo YC/startup: construir una visión grande, pero ejecutarla en fases pequeñas, medibles y muy claras.

Eso significa:

  • tener una North Star clara
  • definir un P0 muy acotado
  • trabajar por PRs pequeños
  • tener criterios de aceptación claros
  • separar arquitectura de reglas operativas
  • evitar scope creep
  • no inventar reglas del negocio
Objetivo: que puedas moverte rápido técnicamente, pero sin comprometer la arquitectura ni asumir cosas que debe confirmar el equipo Santo.

3. Jerarquía de documentos

Para evitar confusión, definimos qué documento manda sobre cuál.

Documento Función
Claude Project Instructions Constitución de largo plazo: visión, arquitectura, reglas, seguridad y roadmap general.

Regla de conflicto: si las Claude Instructions y el PRD/Technical Spec dicen cosas distintas sobre P0, para P0 mandan el PRD v2 y el Technical Build Spec v2.

4. Scope final de P0

P0 no es todo el Santo AI OS. P0 prueba el modelo operativo.

P0 activo:

  • Primario: Corte Santo — Daily Sales Reconciliation.
  • Secundario delgado: Validación Facturas / XML SAT.
  • Secundario delgado: Recibos de Utilidades.

Employee Document Intake no es P0 activo. Puede aparecer en instrucciones anteriores, pero queda como futuro/P3+ salvo que Alonso lo reintroduzca explícitamente.

Mensaje clave: P0 = Corte Santo como workflow principal + XML SAT y Utilidades como pruebas delgadas de reutilización de la foundation.

5. DRY / MECE Skill Registry Rule

Antes de crear un nuevo skill, módulo o script, Dante debe revisar si ya existe algo que cubra esa función o si un skill existente puede parametrizarse.

Debe confirmar:

  1. No existe ya un skill que cubra esta función.
  2. No puede resolverse extendiendo o parametrizando un skill existente.
  3. El nuevo skill no se traslapa con el propósito, inputs u outputs de otro skill.

Si ya existe un skill que puede cubrir el caso, la acción default es extender el skill existente, no crear otro.

Regla: todo skill nuevo debe estar registrado en el Workflow Registry antes del primer commit. Si no está registrado, para Santo AI OS no existe oficialmente.

Esto evita duplicados, scripts aislados y lógica repetida.

6. Progressive Context Loading

santo_context.md debe mantenerse ligero. No debe convertirse en una enciclopedia gigante de toda la empresa.

Debe contener solo contexto general:

  • company overview
  • brand list
  • operating domains
  • fase actual
  • workflow activo
  • reglas generales

El contexto detallado debe vivir por dominio:

  • context_accounting.md
  • context_fiscal.md
  • context_hr.md
  • context_legal.md
  • context_restaurant_ops.md

Cada workflow carga solo lo que necesita. Corte Santo no debe cargar contexto de HR. XML SAT no debe cargar contexto de restaurant ops si no lo necesita.

Por qué importa: esto mantiene el sistema escalable, evita contexto inflado y permite agregar departamentos sin romper la base.

7. Agent Mail en P0

Agent Mail sí es parte de P0. La idea es tener un correo del sistema, por ejemplo os@santo.com, como canal controlado de entrada y notificación.

Agent Mail en P0 debe:

  • recibir correos y adjuntos de workflows
  • crear registros en email_messages
  • ligar correos clasificados a workflow_runs
  • guardar metadata de adjuntos como documents
  • mandar resúmenes, alertas y solicitudes de revisión
  • loggear actividad en events

Agent Mail no debe:

  • ser fuente de verdad
  • aprobar acciones delicadas
  • leer correos no relacionados
  • ignorar silenciosamente correos ambiguos
Regla: un correo ambiguo debe quedar en requires_review, no en ignored.

9. Qué no debe construir todavía

Dante no debe finalizar todavía:

  • thresholds de AMEX, Banorte, efectivo, propinas, Uber, Rappi, etc.
  • rutas definitivas de Drive

  • escritura a Google Sheets para Utilidades

Regla: si falta confirmación del equipo, el sistema debe marcar requires_review, no completed.

10. Operational Addendum

El Operational Addendum será el documento/configuración que salga de las respuestas del equipo Santo. No es otro PRD y no es otra arquitectura.

Su función es convertir respuestas operativas en configuración técnica:

  • drive_folder_map — carpetas, permisos, jerarquía y naming.
  • corte_thresholds — tolerancias y reglas de conciliación.
  • reviewer_map — quién revisa cada tipo de excepción.
  • restaurant_entity_rfc_map — restaurante, razón social, RFC y unidad.
  • agent_mail_routing_rules — cómo clasificar correos del inbox OS.
  • utility_receipt_config — columnas, matching, color/marcado y Sheets scope.
  • xml_sat_config — RFCs, periodos, folders, tipos de XML y fixtures.
Mensaje para Dante: el addendum no cambia la arquitectura; llena la configuración que tú no debes inventar.

11. Regla de no hardcodear

Dante no debe hardcodear:

  • rutas de Drive
  • thresholds
  • reviewers
  • restaurant short codes
  • RFCs
  • email routing
  • source_hash rules
  • columnas de plantillas
  • reglas de excepción

Todo eso debe vivir en configuración, tablas, seed data placeholder o archivos config confirmados.

Traducción simple: no queremos que el sistema funcione porque Dante adivinó cómo opera Santo. Queremos que funcione porque Santo confirmó las reglas.

12. Qué significa requires_review

requires_review es el estado seguro cuando el sistema no tiene suficiente información para decidir.

Debe usarse cuando:

  • falta un threshold
  • falta una carpeta confirmada
  • falta un reviewer
  • un correo no puede clasificarse con confianza
  • un archivo no se puede leer
  • un input no tiene suficiente metadata
  • el sistema detecta ambigüedad
Regla de oro: incertidumbre nunca debe convertirse en completed.

13. Frase corta para explicarlo

Dante, los cambios que hicimos son para que puedas moverte rápido en la base técnica, pero sin inventar reglas de negocio. La arquitectura debe ser reusable, configurable y auditable. P0 prueba el modelo; no intenta construir todo el Santo AI OS desde el día uno.

Si algo no está confirmado por Santo, no lo hardcodees y no lo marques como completed. Déjalo como config pendiente o requires_review.
